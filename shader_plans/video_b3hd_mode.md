# Video B3HD Mode

## Summary
Create a new rotating video mode in `image_video_effects` that sources its video library from the storage API and keeps a list of known video durations. The mode should randomly select segments from across the available videos, using the cataloged lengths to choose valid seek ranges and generate a continuous, high-density playback experience.

## Motivation
- `image_video_effects` already uses `STORAGE_API_URL` and `VIDEO_MANIFEST_URL = https://storage.noahcohn.com/api/songs?type=video`.
- We need a mode that is not just "pick a video", but one that understands each video's duration and uses that data to randomly choose segment start/end boundaries.
- This will enable rotating clips, jump-cuts, and segment-based transitions without guessing or loading full videos blindly.

## Goals
1. Ensure `contabo_storage_manager` exposes video duration metadata in the video catalog.
2. Extend `image_video_effects` to fetch that catalog and preserve duration data.
3. Build a new playback mode that selects random segments from known video lengths.
4. Keep the mode resilient to missing durations by falling back to metadata extraction.

## Storage Manager Changes
### 1. Video index metadata
- Update `image_video_effects/storage_manager/app.py` so `videos/_videos.json` entries include:
  - `id`
  - `name`
  - `filename`
  - `url` or derived stream path
  - `duration` (in seconds, float)
  - `description` / `tags`
  - any existing `rating` / `date`

### 2. API exposure
- The existing `@app.get("/api/songs")` already supports `type=video` and returns video entries from the `video` index.
- Ensure that returned `video` metadata includes `duration` when listing videos.
- If the index is generated by a bucket watcher or upload pipeline, add duration extraction there.

### 3. Upload / update metadata
- Update any video upload/update flows so they can write `duration` into the video metadata and `videos/_videos.json`.
- If a video is ingested as raw file + JSON metadata, make sure `duration` is stored and preserved.

### 4. Streaming route compatibility
- Confirm `/api/videos/{video_id}` supports range requests and `Accept-Ranges: bytes`.
- This matters for seeking to arbitrary start offsets in long videos and for efficient segment playback.
- If not already supported, add range-header handling in the streaming response.

### 5. Optional server-side manifest endpoint
- Consider adding a dedicated endpoint such as `/api/videos/manifest` if `VIDEO_MANIFEST_URL` should become a dedicated video-only catalog.
- For now, using `VIDEO_MANIFEST_URL` via `/api/songs?type=video` is okay, but a dedicated endpoint would be cleaner.

## image_video_effects Changes
### 1. Config / constants
- Keep `VIDEO_MANIFEST_URL` in `src/config/appConfig.ts`.
- Add a new typed constant if needed: `VIDEO_METADATA_URL = `${STORAGE_API_URL}/api/songs?type=video``.

### 2. Content loader expansion
- Extend `src/services/contentLoader.ts` with a `fetchVideoManifest()` path.
- `fetchContentManifest()` currently fetches only `IMAGE_MANIFEST_URL` and local video URLs; split that responsibility out or add a second API call for videos.
- Introduce a new interface such as:
  ```ts
  interface VideoRecord {
    id: string;
    url: string;
    title?: string;
    description?: string;
    tags?: string[];
    duration?: number;
  }
  ```
- Use the API list response to populate `VideoRecord[]` and preserve `duration`.

### 3. Known-duration catalog
- Store a typed list of known videos with duration metadata in app state.
- Example shape:
  ```ts
  interface KnownVideoSegment {
    videoId: string;
    url: string;
    title?: string;
    duration: number;
  }
  ```
- If `duration` is missing from the API, fallback to loading the `HTMLVideoElement` and reading `video.duration` once `loadedmetadata` fires.

### 4. Segment generation logic
- Add a new helper in `src/services/videoSegmentManager.ts` or inside `contentLoader.ts`:
  - `buildVideoSegments(videos, segmentLength, segmentPadding)`
  - `pickRandomSegment(videos, minSegmentLength, maxRandomStart)`
- Use known durations to generate segments like:
  - `start = random(0, Math.max(0, duration - requestedSegmentLength))`
  - `end = Math.min(duration, start + requestedSegmentLength)`
- Keep segment selection safe by never exceeding each video's duration.

### 5. Playback mode integration
- Add a new mode flag such as `videoB3hdMode` or `videoRandomSegmentMode` in the UI state.
- When enabled:
  - Choose a video entry from the catalog at random.
  - Use its known `duration` to pick a valid segment range.
  - Seek the player to the segment start and play until `segmentEnd`.
  - On segment end, choose the next random segment from the same or another video.
- Consider segment selection semantics:
  - Fixed segment duration, e.g. 8-12 seconds.
  - Weighted by video length so longer videos generate more segments.
  - Optionally avoid repeated recent videos by maintaining a short history.

### 6. UI / controls
- Add a control to toggle `B3HD video rotation mode`.
- Add a slider or entry for `segment length` and `transition randomness`.
- Display the currently playing video title + segment range.
- Show `duration` metadata in the video library list if available.

### 7. Fallbacks & resiliency
- If `VIDEO_MANIFEST_URL` returns an empty list, fall back to `FALLBACK_VIDEOS` from `appConfig.ts`.
- If no `duration` is available for a video:
  - load metadata from the video element
  - cache it locally for the session
  - still include the video in the pool only after duration is known
- Ensure remote video files can be seeked; if not, lower the segment length or fall back to full-video playback.

## Implementation Notes
- Prefer catalog-driven rotation, not full playlist shuffle, so the mode can pause and jump cleanly using known durations.
- Keep the segment-selection state separate from shader state; this is a media-layer enhancement, not a shader effect.
- Use existing storage API host and manifest config instead of hardcoding a new endpoint.
- The mode should support both API-driven and local fallback video sources.

## Deliverables

## Success Criteria
- `contentLoader` loads a video catalog including `duration`.
- UI mode picks segments with start/end inside each known video length.
- Playback can rotate through multiple videos without invalid seeks.
- The mode degrades gracefully if duration metadata is unavailable.
